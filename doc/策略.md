# 策略说明文档 (Strategy Documentation)

本文档详细描述了 Urban Renovation 提取项目中使用的各种策略。这些策略定义了 LLM（大语言模型）如何与数据进行交互、提示词的构建方式以及上下文的管理机制。

## 1. 策略概览 (Overview)

目前系统支持以下 5 种提取策略：

| 策略 ID | 名称 | 核心机制 | 上下文管理 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| `single` | Single Turn | 单次提示，一次性输出所有字段 | 独立会话 (Per-paper) | 快速基线测试，低成本 |
| `stepwise` | Stepwise | 分三步交互，引导式提取 | 独立会话 (Per-paper) | 复杂逻辑拆解，提高准确率 |
| `stepwise_long` | Stepwise Long Context | 分三步交互 + 跨论文记忆 | **长上下文 (Cross-paper)** | 测试模型是否能从历史任务中“学习” |
| `cot` | Chain of Thought | 思维链推理 (先思考后输出) | 独立会话 (Per-paper) | 需要深度推理的模糊案例 |
| `reflection` | Self-Reflection | 自我反思与修正 (两轮对话) | 独立会话 (Per-paper) | 减少幻觉，提高合规性 |

---

## 2. 策略详情 (Detailed Strategies)

### 2.1 Single Turn (`single`)

这是最基础的策略，它试图在一个 Prompt 中完成所有工作。

*   **代码位置**: `src/strategies/single.py`
*   **执行流程**:
    1.  初始化一个新的 `ConversationMemory`。
    2.  加载 `System Prompt`（包含所有字段的定义和输出格式要求）。
    3.  构建 `User Prompt`：`[TITLE] ... [ABSTRACT] ...`。
    4.  调用 LLM。
    5.  解析 LLM 返回的 TAB 分隔字符串，提取 4 个字段。
*   **提示词设计**: 使用 `get_single_system_prompt()`，要求模型严格按照 TAB 分隔输出一行结果。
*   **上下文**: 每处理一篇论文，创建一个全新的会话。模型不会记得上一篇论文的处理结果。

### 2.2 Stepwise (`stepwise`)

将复杂的提取任务拆解为三个独立的逻辑步骤，降低模型的认知负担。

*   **代码位置**: `src/strategies/stepwise.py`
*   **执行流程**:
    1.  初始化一个新的 `ConversationMemory`。
    2.  **Step 1**: 询问“是否属于城市更新研究？”。LLM 回答 1/0。
    3.  **Step 2**: 询问“是否为空间研究？”。LLM 回答 1/0。
    4.  **Step 3**: 询问“空间等级和具体描述”。LLM 回答 TAB 分隔的后两个字段。
    5.  合并三步的结果。
*   **提示词设计**: 使用 `get_step_system_prompt()` 和 `get_step_prompt(step_num)`。
*   **上下文**:
    *   **论文内 (Intra-paper)**: 保留。在处理同一篇论文的 3 步过程中，历史对话被保留，模型知道之前的回答。
    *   **论文间 (Inter-paper)**: **不保留**。处理完一篇论文后，Memory 被销毁。

### 2.3 Stepwise Long Context (`stepwise_long`)

逻辑与 `stepwise` 相同，但引入了长上下文机制，允许模型在处理多篇论文时保持“记忆”。

*   **代码位置**: `src/strategies/stepwise_long.py`
*   **核心差异**:
    *   它**复用**同一个 `ConversationMemory` 对象，而不是每次创建新的。
    *   使用 `_get_or_create_memory` 方法管理生命周期。
    *   只有当当前 Token 数超过阈值（默认 128k 的 90%）时，才会强制重置 Memory 并开启新会话。
*   **目的**: 观察模型是否会因为“看到”了之前的论文（及其判定结果）而产生 In-context Learning（上下文学习）效应，从而提高后续提取的一致性。
*   **风险**: 如果模型之前犯错，错误的上下文可能会误导后续判断（累积误差）。

### 2.4 Chain of Thought (`cot`)

通过强制模型显式输出推理过程，来提高复杂判断的准确性。

*   **代码位置**: `src/strategies/cot.py`
*   **执行流程**:
    1.  初始化独立会话。
    2.  发送 Prompt，其中包含特殊指令：要求先在 `<thinking>` 标签中输出推理步骤，然后再输出最终结果。
    3.  LLM 返回包含推理和结果的文本。
    4.  **解析器**: 专门的解析逻辑会忽略 `<thinking>...</thinking>` 部分，只提取最后一行有效的 TAB 分隔数据。
*   **提示词设计**: 使用 `get_cot_system_prompt()`，该 Prompt 继承自 Single Prompt 但覆盖了输出要求部分。

### 2.5 Self-Reflection (`reflection`)

模拟人类的“检查”机制，让模型自己审视初步答案。

*   **代码位置**: `src/strategies/reflection.py`
*   **执行流程**:
    1.  **Round 1**: 发送常规 Single Prompt。获取 `Response 1`（初步答案）。
    2.  **Round 2**: 发送 Critique Prompt（“请回顾你的答案，严格对照定义检查... 如果错误请修正”）。
    3.  获取 `Response 2`（修正后的答案）。
    4.  解析 `Response 2` 作为最终结果。
*   **提示词设计**:
    *   Round 1: 同 `single`。
    *   Round 2: `get_reflection_critique_prompt()`，强调定义的一致性。

## 3. 提示词模式 (Shot Modes)

所有策略都支持以下三种 Shot 模式，由 `src/prompts.py` 中的 `PromptGenerator` 控制：

1.  **Zero-shot**: 仅提供任务定义和规则，不提供任何示例。
2.  **One-shot**: 在 System Prompt 中附加 **1 个** 完整的输入输出示例。
3.  **Few-shot**: 在 System Prompt 中附加 **3 个** 代表性示例（通常覆盖正样本、负样本、边缘情况）。

**注意**:
*   对于 `stepwise` 策略，示例也是分步展示的（Step 1 Q&A, Step 2 Q&A...）。
*   对于 `cot` 策略，示例应包含 `<thinking>` 过程（目前代码中复用了 Single 的示例，可能需要针对性优化）。

## 4. 扩展指南 (Extensibility)

如果您需要添加新的策略（例如 `ensemble` 投票策略）：

1.  **创建策略类**:
    *   在 `src/strategies/` 下新建文件（如 `ensemble.py`）。
    *   继承 `ExtractionStrategy` 基类。
    *   实现 `process(self, title, abstract)` 方法。

2.  **注册策略**:
    *   在 `src/strategies/registry.py` 中导入新类。
    *   在 `_strategies` 字典中添加映射：`"ensemble": EnsembleStrategy`。

3.  **更新主程序**:
    *   修改 `scripts/main.py` 中的 `select_strategy()` 函数，添加新选项。

4.  **添加提示词 (可选)**:
    *   如果新策略需要特殊的 System Prompt，请在 `src/prompts.py` 中添加相应的方法（如 `get_ensemble_system_prompt`）。
