# 403抓包诊断指南（Windows）

## 目标
- 针对 OpenAI 兼容入口返回 403 的场景，建立可复现证据链。
- 输出包含：请求路径、模型字段、关键响应头、错误体摘要、根因归类。

## 执行步骤
1. 启动本机代理抓包工具并开启 HTTPS 解密。
2. 在终端设置代理后执行最小化探测脚本：
   - `python scripts/debug_probe_llm.py --proxy http://127.0.0.1:8888`
3. 在抓包工具中过滤目标域名，定位 `POST /chat/completions` 请求。
4. 记录以下字段：
   - Request: method、host、path、Authorization 形态、body.model
   - Response: status、headers(request-id/x-request-id)、body.error
5. 运行主流程进行二次对照，确认是否同样 403：
   - `python scripts/main.py --task urban_renewal --limit 1`
6. 对照最小脚本与主流程的报文差异并归类根因。

## 判定矩阵
- 403 + `model_not_found` / `forbidden_model`
  - 含义：Key 无该模型权限或模型名无效。
  - 处置：更换已授权模型名，或开通模型权限。
- 403 + `invalid_api_key` / `insufficient_user_authentication`
  - 含义：Key 不可用、被封禁、或网关未正确透传鉴权。
  - 处置：更换有效 Key；核对网关鉴权转发策略。
- 403 + 网关自定义错误码（非上游模型错误）
  - 含义：第三方网关白名单、IP 限制、账号风控或路径策略拦截。
  - 处置：联系网关方放行；改用官方兼容端点回归验证。
- 404/405 且路径异常
  - 含义：`base_url` 与 SDK 组合后路径不正确。
  - 处置：修正 `LLM_BASE_URL` 至供应商要求格式。

## 回归标准
- 最小化探测脚本成功返回内容且非 403。
- 主流程 `--limit 1` 成功完成并产出结果。
- 诊断输出中包含 request-id 或等价追踪字段，便于后续追溯。
